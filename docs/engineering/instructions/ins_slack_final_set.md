# INS: 「余裕あり」判定を最終セット基準に変更(DEC-007)

- 作成者: PM (Claude web)
- 日付: 2026-07-20
- 対象: `src/engine/progression.ts`(`suggestWeightReps`)
- 関連: ISS-013b(余裕ありフィードバック)、DEC-007、SSOT更新要
- 備考: DEC-006とは別PRとすること(対象ファイル・関心が異なる)

---

## 0. 背景と決定事項(DEC-007)

現行の2ステップ増量判定は `recordedSets.some((s) => s.hadSlack === true)` であり、**どのセットに付いた「余裕あり」でも発火する**。しかし1セット目は累積疲労がない状態なので余裕があって当然であり、負荷の適正を語る信号にならない。負荷が本当に軽いことを示すのは、疲労が乗った**最終セット**で上限レップに到達し、なお余裕がある状態である。

**決定(Eiichi指摘・承認済み):**
2ステップ増量のトリガーとなる「余裕あり」は、その種目の**最後に記録されたセット**の `hadSlack` のみで判定する。

### 補足(変更しないもの)
- `atFailure`(限界でした)の `some` 判定は現行維持。途中セットの限界も「増量保留」の正当な信号であり、余裕とは非対称でよい。
- データモデル(`hadSlack?: boolean` セット単位)とUI(セットごとのトグル)は変更しない。エンジンの読み方だけを変える。
- レップ先行の原則(全セット達成+上限到達が増量の前提)は不変。

---

## 1. 仕様

1. `suggestWeightReps` 内の判定を変更:
   - 変更前: `const anySlack = recordedSets.some((s) => s.hadSlack === true)`
   - 変更後: 最後に記録されたセット(`recordedSets` の末尾要素)の `hadSlack === true` を判定に使う。変数名も `lastSetSlack` 等、意味が伝わる名前に改める。
2. `recordedSets` は `reps !== undefined` でフィルタ済みの配列であるため、「最後に記録されたセット」= フィルタ後配列の末尾でよい。
3. ドキュメンテーションコメント(関数上部の「余裕あり付きなら2ステップ増量」の記述)を最終セット基準である旨に更新する。

## 2. テスト(`src/engine/progression.test.ts`)

1. 最終セットに `hadSlack: true` + 全セット上限到達 → 2ステップ増量(連続上限未満のとき)
2. **1セット目のみ** `hadSlack: true`(最終セットはfalse/未設定)+ 全セット上限到達 → 通常の1ステップ増量
3. 最終セット `hadSlack: true` だが上限未到達 → 増量しない(レップ先行の回帰)
4. 既存の `anySlack` 前提のテストケースは新仕様に合わせて更新
