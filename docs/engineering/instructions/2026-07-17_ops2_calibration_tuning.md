# Instruction: 定常運転 #2 — キャリブレーション係数調整+「余裕あり」フィードバック

- Priority: 🟡
- 発行: PM / 2026-07-17
- 前提: main最新
- 対応Issue: ISS-013(実地検証結果: 初期提案11.5kgは「各セットあと数回余裕」= 保守的すぎ)

## 背景(実測データ)

Eiichiの実地検証: バーベルベンチ45kg×7(推定1RM 55.5kg)→ 換算0.21 → ダンベルプレス11.5kg提案。
実際の体感は「全セット完遂+各セットあと数回の余裕」。上限到達だが限界には遠い状態で、
現行エンジンはこれを1ステップ増量でしか反映できず、適正重量への収束が遅い。

## 要件

### 1. 換算係数の引き上げ(ISS-013a)
- `src/constants/strength.ts` の水平プレス係数 0.21 → **0.23**
- 他パターンの係数も同比率(+約10%)で見直し、根拠コメントを更新(実測1点による補正である旨を明記)
- 既存の受け入れテスト(45kg×7→10〜12.5kg帯)を新係数の期待値に更新

### 2. 「余裕あり」フィードバック(ISS-013b)
- セット記録UIに「💨余裕あり」ワンタップを追加(既存「🔥限界でした」の対になる配置・トーン)
- エンジン反映: 上限レップ到達 かつ 余裕ありフラグ → 次回**2ステップ増量**(通常は1ステップ)。
  余裕あり単独(上限未達)では増量しない(レップを先に伸ばす原則は維持)
- 連続適用の暴走防止: 2ステップ増量は連続2回まで、以降は通常進行(定数化)
- ユニットテスト: 「上限+余裕→2ステップ」「上限のみ→1ステップ」「余裕+未達→増量なし」「連続上限3回目→1ステップ」

## 備考

- 既存ログがある種目はダブルプログレッション優先(係数変更の影響は「ログなし種目の初期値」のみ)。
  つまり既にトレ済みの種目の提案が急に跳ねることはない——handoffでこの点を確認・明記すること
