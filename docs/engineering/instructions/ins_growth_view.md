## 0. モック突き合わせ(炉心§0方式)

実装前に `growth_viz_concepts.md` と実アプリ仕様の差分を確認し、モック側の仮定が実データ・実装と食い違う点があれば本INSの解釈を優先し、判断に迷うものはハンドオフに記録すること。既知の確定事項:

- e1RM算出式は **Epley(レップ12クランプ)** で確定: `e1rm = weight * (1 + min(reps, 12) / 30)`。モック内の仮置き注記は本決定で上書き
- 色スケールはモックの6段階閾値表を **絶対値固定** で採用。適応スケーリングは実装しない(ISS起票のみ、§5参照)

## 1. 成長算出仕様(コア)

1. **算出関数の隔離**: e1RM計算・基準種目選定・成長率算出は `src/engine/growth.ts`(新規)に純関数として集約。UIからはこのモジュールのみを参照
2. **セッションe1RM(種目単位)**: そのセッション内の全記録セットについてEpley値を計算し、最大値を採用
3. **基準種目(部位単位)**: 表示期間内でセッション数が最多の種目。同数の場合は直近セッションが新しい方。部位の成長はこの基準種目の中でのみ比較する(種目をまたいだe1RM比較は行わない)
4. **成長率**: `(期間内最新セッションe1RM − 期間内最古セッションe1RM) / 期間内最古セッションe1RM`。**変化率チップ・グラフの表示は実測%**(その期間で実際に伸びた率)。**人体図の色エンコーディングのみ月換算**(90日表示時は×30/90)——期間を切り替えても色の意味が変わらないため。`growth_viz_concepts.md` の正規化方針に準拠
5. **データ不足判定**: 表示期間内で基準種目のセッションが3回未満の部位は「冷えた鉄」表示(変化率・推移非表示、「あと n 回」表示)。モックの定義を踏襲
6. **色スケール**: `growth_viz_concepts.md` の閾値表を `GROWTH_HEAT_SCALE` として定数化(月換算% → 色)。将来差し替え前提のコメント付き

## 2. 成長ビュー(4a完成版準拠)

1. 人体図(FRONT/BACK)ヒーロー: FreshnessBodyMapのジオメトリを再利用し、色を成長スケールで塗る
2. 期間切替 30日/90日
3. 部位カード: 変化率チップ。タップでe1RM推移グラフ(Recharts・データ点=セッション)
4. グラフカードタップでフルスクリーン: 拡大グラフ+セッション履歴(日付・e1RM・前回差)。データ不足部位はタップ無効
5. ナビゲーション: ダッシュボードの成長カードから遷移(§3)。戻る導線を明確に

## 3. ダッシュボード組み込み(4b準拠)

1. 成長カード: ミニ人体図+上位3部位の変化率要約。タップで成長ビューへ
2. 回復予測スロット: 最下部固定。**DEC-010の `hoursUntilRecovered` を再利用**し、二重実装しない。tuningオーバーライド反映もそのまま乗ること
3. 色の意味分離(成長=溶鉄の彩度/回復=鈍色)はモックのトークン指定に従う

## 4. テスト

1. Epley式とレップ12クランプの境界(reps=12, 13)
2. セッションe1RM=セット最大値
3. 基準種目選定(最多/同数タイブレーク/期間で変わるケース)
4. 成長率の月換算(30日/90日)
5. データ不足判定(2回→非表示、3回→表示)
6. 色スケール写像(閾値の境界値)
7. 既存142本の回帰維持
