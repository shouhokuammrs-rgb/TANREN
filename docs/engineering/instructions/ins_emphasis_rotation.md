## 0. 背景と決定事項(DEC-012)

「大胸筋の上部・下部のような粒度で管理したい」という要望に対し、回復・選択・ゴールの**管理単位は現行7部位のまま維持**する(部位内区分は生理学的に回復を分離できず、e1RMも区分単位では成立しないため、細分化は疑似精度になる)。代わりに、発達の偏りの実体である**種目選択の偏り**を解消する。

現状の問題: `candidatesByMuscle` の並びが決定的(コンパウンド→実績→ID)で、同一部位で毎回ほぼ同じ種目が選ばれる。ローテーションが存在しない。

**決定(Eiichi承認済み):**
種目に「強調区分」タグを付け、メニュー生成が同一部位内で強調をローテーションする(前回と違う強調の種目を優先)。管理単位・回復モデル・成長算出は変更しない。

## 1. データモデル

1. `ExerciseEmphasis` 型を新設(部位内の強調区分):
   - 胸: `'upper' | 'mid' | 'lower'`
   - 肩: `'front' | 'side' | 'rear'`
   - 腕: `'biceps' | 'triceps'`
   - 脚: `'quad' | 'ham_glute'`
   - 型としては上記の文字列unionを1つにまとめてよい
2. `Exercise` に `emphasis?: ExerciseEmphasis` を追加。**undefined = 中立**(ローテーション評価の対象外だが選択は可能。背中・腹・尻はv1では中立のままでよい)
3. マスタ(`constants/exercises.ts`)へのタグ付け。種目名から明確なもののみ:
   - インクライン系=upper、フラットのプレス/フライ/プッシュアップ=mid、デクライン系=lower
   - ショルダープレス/パイクプッシュアップ=front、サイドレイズ/アップライトロウ=side、リアデルトフライ=rear
   - カール系=biceps、トライセプス系/ディップス=triceps
   - スクワット/ランジ系=quad、RDL等ヒンジ系=ham_glute
   - 迷う種目は無理にタグ付けせず中立に残し、ハンドオフに列挙すること
4. 既存DBへの反映はマスタ同期の既存機構に従う(ユーザー登録値を壊さないこと)

## 2. エンジン(選択ロジック)

1. `EngineContext` に `recentEmphasis?: Map<MuscleGroup, ExerciseEmphasis[]>` を追加(**新しい順**。DB層が直近のセッション履歴から組み立てる。対象範囲は部位ごとの直近3セッションを基本とし、定数化)
2. `candidatesByMuscle` のソートを変更。優先順位は:
   1. コンパウンド優先(現行維持・最上位)
   2. **強調ローテーションスコア**: `recentEmphasis` に出現していない強調が最優先、出現している場合は古いものほど優先(LRU)。中立(undefined)種目はこの層では同順位として扱う
   3. 実績あり優先(現行)
   4. ID順(現行・決定性の担保)
3. `alternativesFor`(入れ替え候補)にも同じソートを適用し、生成と入れ替えで一貫させる
4. エンジン純粋性は維持(履歴の読み出しはDB層、エンジンはctx経由)

## 3. UI

1. `MenuItem` に強調を載せ、WorkoutPage の種目行に小さなチップで表示(「上部」「サイド」等。ラベルは `copy.ts` に集約)。中立種目はチップなし
2. これ以上のUIは作らない。「最近の強調カバレッジ」可視化は**ISS-017として起票のみ**(将来、成長ビュー拡張時にデザイン探索と合わせて検討)

## 4. テスト

1. LRUローテーション: 前回upperなら次はmid/lowerの種目が上位に来る
2. 未出現の強調が最優先になる
3. コンパウンド優先がローテーションより上位であること(インクラインフライがフラットプレスを追い越さない)
4. 中立種目の順位が現行と変わらない(背中・腹の回帰)
5. `alternativesFor` と生成の並びが一貫する
6. 既存158本の回帰維持

## 5. SSOT更新指示

- **ISS-017**: 強調カバレッジの可視化(部位詳細での偏り表示)。将来のデザイン探索と合わせて検討
